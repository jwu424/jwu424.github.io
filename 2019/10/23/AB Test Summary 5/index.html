Created 2019-10-23| Updated 2019-10-24|AB Testing

This is a summary of Unexpected Results in Online Controlled Experiments. There are several lessons that is worthy share.

Introduction

Any figure that looks interesting or different is usually wrong。 —- Twyman’s law

Lesson1: BROWSER REDIRECTS

A very common and practical mechanism used to implement an A/B test is to redirect the treatment to another page. However, A/A test fails. And the version with redirect significantly underperformed the other version.

Implementation:

If the randomization function determines that the user should be in Control, the page is displayed; if the randomization shows that the user should be in Treatment, a browser redirect is done by using the http-equiv=”REFRESH” meta tag in HTML.

Reason:

  • Extra redirect cost more time and the slowdowns have significant impact on metrics.
  • Bots. Different robots will handle redirects differently: some may not redirect, some will tag this as a new page worthy of deep crawling, etc. Thus, our experiments are biased.
  • Redirects are asymmetric. When users are redirected to the treatment page, they may bookmark it or pass a link to their friends. Bots might add this new page to their index for crawling. And the experiment has contamination.

Lessons:

  • avoid redirects in implementations and prefer a server-side mechanism that generates HTML.
  • run an A/A’/B’ test.
    • comparison between A’ and B’ is therefore fair.
    • the difference between A and A’ gives us an idea of the impact of the redirect on key metrics.

Lesson2: EXPOSURE CONTROL

In several situations, we saw surprising results that were traced to bad exposure control.

Ex1:

  • Test a new version US homepage.
  • New version doesn’t have reverse-IP lookup, but the old version has.
  • Most non-US IPs are redirected to the Home pages of other countries.
  • The results were highly biased because the population of users from non-US IPs was much higher in the Treatment than in the Control.

Ex2:

  • In a Bing experiment, a misconfiguration caused all Microsoft users to always see Control.
  • This created enough of a bias to skew results.

Lessons:

  • Run A/A tests that resemble the final setup as close as possible.
  • down and slice the data by common attributes, such as country and browser.
  • Large differences may hint at improper exposure control.

Lesson3: EXPOSURECONTROL

When run normal A/B test, the highest overall power is achieved when the population split is 50%/50%.

In practice, treatments may need to run at lower percentages. Like in ramp-up of an experiment, it starts at very lower percentages. But if it is a large site, it may have enough power.

if multiple disjoint experiments need to be run, they may share a control, which would be larger; if one is interested in running comparisons between different treatments to the Control, a larger control provides more power.

Ex:

  • Run A/A test at 90%/10%, it fails consistently.
  • It turns out that a bounded resource is the cause.
  • In this case an LRU (least- recently-used) cache was used, and the entries for the Control and Treatment were disjoint. Because the experiment ran as a 90%/10% experiment, the Control had significantly more entries in the LRU cache, leading to a higher cache-hit ratio and thus better performance, impacting the user experience and leading to better metrics for the control.

Lessons:

  • Run A/A tests and be vigilant about measuring performance.

Lesson4: BROWSER DIFFERENCES

Ex:

  • Test whether the link should open Hotmail in a new window rather than in place.
  • Find a problem: the percentage of users who clicked on the Hotmail link is significant different, which should be the same. Since it happens before the difference of the webpage.
  • Reason:
    • Clicks are instrumented by using a web beacon or web bug, which are lossy.
    • it turns out that by opening the destination in a new window, the beacon’s reliability improved significantly for non-IE browsers and hence the delta.

Lessons:

  • Investigate anomalies seriously.
  • Proactively drill-down by key attributes, such as browser family (based on user-agent) and geography (based on reverse IP).

Lesson5: LONG-TERM OEC

EX:

  • Campaigns that sent e-mails to users, introducing them to products they may be interested in.

  • The OEC for the campaigns was based on purchases whose sessions were referred by the

    e-mails.

  • a large number of e-mails to be sent and customers complained.

Fix:

  • penalized for unsubscribes, as these customers are no longer targetable in future campaigns.

Lessons:

  • Always including long-term part or customer lifetime value in the OEC.

Lesson6: MONITORING SYSTEMS

Ex:

  • Compare two checkout pages.
  • New page has a lower conversion rate but the number of page views per user is higher.
  • New page adds a monitor system and the system was designed such that if the click failed, it would try multiple times before raising an alarm.
  • It turned out that with the new Treatment design, the “click” action from the monitoring system did not work and it made many retries, reducing the click-through rate for the Treatment.

Lessons:

  • take bots and monitoring systems into account.
  • Monitoring systems can create large skews if unaddressed.

Lesson7: UNPLANNED DIFFERENCE BETWEEN VARIANTS

Ex:

  • an experiment comparing the ordering of headlines on the MSN Home Page
  • randomly placed headlines had a 2% increase in clicks and was highly significant
  • Drill down to hourly data and find the pobelm.

Lessons:

  • Experimental control is critical. Keep everything constant except the thing you want to test.
  • Drill-down by time to look at hourly data. Regularly show hourly plots for sanity checks to detect such anomalies.
  • Use screen scrapers to save screen shots of the pages being experimented on a regular basis in order to allow debugging of surprises.

Lesson8: SIMPSON’SPARADOX

Ex:

  • Treatment was worse but if plot each day, treatment groups were always better.
  • The experiment was a ramp-up.
  • The metric follows the usual pattern of clicks per user being lower on the weekends than weekdays, so the last two days had fewer clicks per user than the average, but the Treatment effect was still positive.
  • However, since the Treatment had a much larger percentage on the last two days, the clicks per user on those days carried larger weight with the Treatment mean making the Treatment look worse than the Control.
  • Simpson’s paradox.

Lessons:

  • Beware the potential impact of Simpson’s paradox
  • Using special analyses options you can take to make sure Simpson’s paradox doesn’t impact your results.
  • Or any ramp-up period must be completed prior to the start of the experiment.

Lesson9: SHOW WHAT WANT

Ex:

  • test a redesigned homepage that looked more modern, was a cleaner design and had fewer links to distract from the primary objective, getting users to click on the buttons to download a version of Microsoft Office for trial or purchase.
  • But the number of clicks decrease 64%.
  • Reasons: the new version showed the actual price whereas the old one didn’t show but ‘Try free’.
  • This does not mean there are more purchases, but rather that the conversion rate during the purchase pipeline may be different with the Treatment sending more qualified users to the pipeline.

Lessons:

  • Always get information on the ultimate action you want the user to take.

Lesson10: BEWARE THE ASSUMPTION

Ex:

  • Test two metric for A/A test.
  • One metric had independent units, whereas the another one had positively correlated units.
  • Used the standard statistical formula for standard deviation.
  • The first one was 5% significant.
  • The second one was 30%, for the standard deviation was underestimated by two-thirds

Fix:

  • Use bootstrapped estimate

Lessons:

  • Beware of classical statistical formulas that assume independence.
